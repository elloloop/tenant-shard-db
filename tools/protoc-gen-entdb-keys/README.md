# protoc-gen-entdb-keys

`protoc-gen-entdb-keys` is a `protoc` plugin that reads
`(entdb.field).unique = true` annotations from your `.proto` files and
emits typed `UniqueKey` sidecar files for the EntDB SDK in both Python
and Go. For every input `foo.proto` the plugin writes `foo_entdb.py`
and `foo_entdb.go` next to the standard `protoc-gen-python` /
`protoc-gen-go` outputs. The generated tokens are the only way to
perform a `get_by_key` lookup in the EntDB SDK, which is how the SDK
eliminates stringly-typed key references. See
[`docs/decisions/sdk_api.md`](../../docs/decisions/sdk_api.md) for the
authoritative SDK surface ADR.

## Build

```
make build
```

Produces `bin/protoc-gen-entdb-keys`. To install into `$GOBIN`:

```
make install
```

## Invocation

```
protoc \
  --python_out=. \
  --go_out=. --go_opt=paths=source_relative \
  --entdb-keys_out=. \
  --plugin=protoc-gen-entdb-keys=./bin/protoc-gen-entdb-keys \
  schema.proto
```

Given a `schema.proto` with:

```proto
message Product {
  option (entdb.node) = { type_id: 201 };
  string sku     = 1 [(entdb.field).unique = true];
  string barcode = 2 [(entdb.field).unique = true];
}
```

the plugin emits `schema_entdb.py`:

```python
# Code generated by protoc-gen-entdb-keys. DO NOT EDIT.
from entdb_sdk import UniqueKey

class ProductKeys:
    sku     = UniqueKey[str](type_id=201, field_id=1, name="sku")
    barcode = UniqueKey[str](type_id=201, field_id=2, name="barcode")
```

and `schema_entdb.go`:

```go
// Code generated by protoc-gen-entdb-keys. DO NOT EDIT.
package shop

import "github.com/elloloop/tenant-shard-db/sdk/go/entdb"

var (
    ProductSKU     = entdb.UniqueKey[string]{TypeID: 201, FieldID: 1, Name: "sku"}
    ProductBarcode = entdb.UniqueKey[string]{TypeID: 201, FieldID: 2, Name: "barcode"}
)
```

Messages without `(entdb.node)` are skipped entirely. Node messages
with zero unique fields are also skipped (no empty classes or empty
`var` blocks). Go variable names are `<MessageName><FieldPascalCase>`
with common initialisms (`SKU`, `ID`, `API`, `URL`, `HTTP`, `UUID`, …)
promoted to upper-case per Go style.

## Type mapping

| Proto                                | Python  | Go        |
| ------------------------------------ | ------- | --------- |
| `string`                             | `str`   | `string`  |
| `int32`, `sint32`, `sfixed32`        | `int`   | `int32`   |
| `uint32`, `fixed32`                  | `int`   | `uint32`  |
| `int64`, `sint64`, `sfixed64`        | `int`   | `int64`   |
| `uint64`, `fixed64`                  | `int`   | `uint64`  |
| `float`                              | `float` | `float32` |
| `double`                             | `float` | `float64` |
| `bool`                               | `bool`  | `bool`    |
| `bytes`                              | `bytes` | `[]byte`  |

`(entdb.field).unique` may only be applied to a **scalar, singular,
non-oneof, non-optional** field. Messages, enums, groups, maps,
repeated fields, oneof fields, and proto3 `optional` fields are
rejected with a clear error message that names the offending
`Message.field`. Unique constraints on non-scalar values do not have a
meaningful indexable representation and the plugin refuses to guess.

## Test

```
make test
```

The test suite covers:

- Golden-file comparison for the sample `Product`/`Order` proto
- All 15 supported scalar types via `types.proto`
- Initialism handling (`sku`, `external_id`, `api_key`, `http_url`,
  `order_number`, `customer_uuid`)
- Rejection of non-scalar unique fields
- Skipping non-node messages and node messages with no unique fields
- Unit tests for the `snake_case → PascalCase` helper and the type
  mapping table
